# Workbench UX 实施方案

## 目标定位

Agent Workbench 第一阶段要从工程执行面板改成对账运营工作台：

```text
上传资料 -> 生成账单 -> 问 Agent -> 处理建议 -> 归档经验
```

用户默认看到业务动作，不看到内部执行细节。真实账单引擎、任务编号、事件流、对象存储、原始请求等信息仍然保留，但只放在“查看技术详情”里。

## 用户可见词表

| 内部/旧说法 | 用户可见说法 |
| --- | --- |
| Athena 出账 | 生成账单 |
| Agent 调查 | 问 Agent / 对账 Agent |
| Job | 任务 |
| Agent Session | 本次对话 / 历史经验 |
| Artifact / 产物 | 结果文件 |
| Payload | 技术详情 |
| Object Storage / S3 | 资料归档 |
| Stream / Events | 运行记录 |

| Human-in-loop / 人工介入 | 补充信息 |
| Config version | 计费口径 |
| 审批 | 处理建议 / 应用建议 / 忽略建议 |

## Agent 页面

Agent 是一个可以实时对话的助手，只是默认带上了本次对账上下文。页面应以对话为主，而不是以运行控制台为主。

推荐结构：

```text
左侧：本次资料 + 历史经验
中间：实时对话
右侧：结论与建议
```

左侧展示 Agent 已知道什么：

- 账期、渠道、供应商。
- 当前账单和上传资料。
- 计费口径。
- 历史经验入口。

中间对话区：

- 输入框固定在底部。
- 默认提示“我已经带上本次账单、供应商资料和计费口径，你可以直接问我差异原因、折扣口径或下一步怎么处理。”
- 用户可以随时追问、补充资料、继续对话。

右侧结果区：

- 发现的问题。
- 影响金额。
- 可能原因。
- 建议动作。
- 可保存的经验。
- 相关结果文件。

技术信息只出现在折叠项里，例如原始事件、工具结果、请求详情。

## 历史经验

历史 Session 必须完整保留。用户侧把它称为历史经验或历史对话，因为这些记录包含大量可复用判断。

历史经验列表需要支持：

- 搜索。
- 按供应商、账期、标签筛选。
- 收藏。
- 继续对话。
- 引用到当前任务。

打开历史经验后应展示：

- 原始对话。
- 用户补充信息。
- Agent 结论。
- 相关资料。
- 最终处理动作。
- 可复用经验摘要。

## 生成账单页面

生成账单主表单只展示高频业务字段：

- 账期。
- 范围。
- 供应商。
- 计费口径。

缓存、汇率、明细导出、输出目录、原始参数等放入高级设置。页面不强调 Athena、CLI、Worker 等执行细节。

## 处理建议

不做审批流程。建议处理动作包括：

- 应用建议。
- 忽略建议。
- 重新生成账单。
- 保存为经验。

后台可以继续保留版本、处理人、处理时间和前后差异，保证追溯，但不要把它做成审批队列。

## 真实部署保护

真实 ECS 环境不能让 Web/API 直接无限执行长任务。第一阶段必须具备最小保护：

- 任务进入队列或被并发保护拦截。
- 账单任务默认同时 1 个。
- Agent 核验默认同时 1 个。
- 单任务运行超时。
- 上传大小和临时目录大小限制。
- 任务完成后清理本地临时文件。
- 结果统一归档。
- 大范围查询保护。

## 并行实施线

| 工作线 | 首批交付 |
| --- | --- |
| UX 与文案 | 清理导航、按钮、标题和主流程脏术语 |
| Agent 界面 | 三栏实时对话界面，保留历史经验入口 |
| 历史经验 | Session 列表重命名、搜索/引用/继续对话入口 |
| 生成账单 | 隐藏 Athena/CLI 细节，压缩主表单 |
| 建议处理 | 去审批化，改为应用/忽略/保存经验 |
| 运行保护 | 后端并发限制、队列状态、超时入口 |
